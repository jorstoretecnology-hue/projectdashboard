🔴 Problemas críticos
🚨
9 archivos SQL de reparación en el repo
NUKE_AND_REBIRTH.sql · REPARACION_ROBUSTA.sql · FIX_REGISTRATION_V2.sql · ...
Crítico
▼
Este es el hallazgo más preocupante del repo. Tener 9 archivos de reparación manual del tenant_id en la raíz del proyecto significa que el sistema de autenticación multi-tenant se ha roto repetidamente y se ha parcheado de forma manual.

El archivo NUKE_AND_REBIRTH.sql es la señal más clara: en algún momento la base de datos se rompió tan mal que hubo que reconstruirla desde cero.

Esto NO es un problema de código — es un problema de que el esquema de multi-tenancy está frágil. La raíz del problema probablemente es que tenant_id se está sincronizando en múltiples lugares (JWT metadata + tabla profiles + raw_app_meta_data) y cuando uno falla, el sistema colapsa.
La solución
-- El tenant_id debe vivir en UN solo lugar: la tabla profiles.
-- El JWT metadata es cache. El source of truth es la DB.

-- Función get_current_user_tenant_id() CORRECTA:
CREATE OR REPLACE FUNCTION get_current_user_tenant_id()
RETURNS uuid AS $$
  SELECT tenant_id
  FROM profiles
  WHERE id = auth.uid()
  LIMIT 1;
$$ LANGUAGE sql SECURITY DEFINER STABLE;

-- Mover estos archivos a /archive/ y crear migraciones limpias.
-- Nunca más parchar manualmente en producción.
Acción inmediata
1. Mover todos los REPARACION*.sql a /docs/archive/
2. Crear una migración limpia y definitiva en /supabase/migrations/
3. Documentar exactamente cuándo y por qué ocurrían los fallos para no repetirlos.

Arquitectura 2 niveles — no soporta multi-sede ni multi-negocio
tenants.industry_type es un campo único · No hay tabla locations
Crítico
▼
El schema actual es: User → Tenant. Un tenant tiene UN industry_type.

Esto significa que si tu cliente del restaurante quiere agregar las cabañas y el glamping, necesitaría crear un tenant completamente nuevo, con login separado, pagar por separado, y no habría forma de ver reportes unificados.

Tampoco soporta múltiples sedes del mismo negocio — el taller con 2 locales necesitaría 2 cuentas distintas.

La buena noticia: el schema tiene industry_type como campo en tenants, lo que significa que ya piensas en multi-industria. Solo falta agregar la capa de locations.
Lo que hay que agregar (no reescribir)

CREATE TABLE locations (

);

-- Agregar location_id opcional en tablas operativas
ALTER TABLE inventory_items
ADD COLUMN location_id uuid REFERENCES locations(id);

-- Crear location "principal" automáticamente al crear tenant
-- Así el código existente sigue funcionando sin cambios.
La estrategia: NO reescribir todo. Agregar locations como capa opcional. El código actual con tenant_id sigue funcionando. Se agrega location_id donde hace falta (inventario, ventas, reservas) como nullable primero.

Middleware no protege rutas — delega todo al cliente
src/middleware.ts — solo llama updateSession, no verifica roles
Crítico
▼
El middleware actual solo refresca la sesión pero no verifica rutas protegidas. Las rutas definidas como "protectedRoutes" y "superAdminRoutes" están comentadas pero nunca usadas. Cualquier usuario podría intentar acceder a /superadmin y la protección depende únicamente del frontend.
Fix en middleware.ts
export async function middleware(request: NextRequest) {
  const response = await updateSession(request)
  const { pathname } = request.nextUrl
  const supabase = await createClient()
  const { data: { user } } = await supabase.auth.getUser()

  // Ruta superadmin sin usuario SUPER_ADMIN → redirect
  if (pathname.startsWith('/superadmin')) {
    const role = user?.app_metadata?.app_role
    if (role !== 'SUPER_ADMIN') {
      return NextResponse.redirect(new URL('/dashboard', request.url))
    }
  }

  return response
}
